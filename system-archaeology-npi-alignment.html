Overview

Why this matters to your business

Delivery risk often comes from uncertainty: uncertainty about how the system behaves, which assumptions it relies on, and whether teams share the same understanding. When that uncertainty is left unresolved, problems appear late as integration delays, rework, and “unexpected” test failures.

Common symptoms

  • System behaviour differs from what stakeholders believe is implemented.
  • Project deliverables (software, wiring, CAN, documents) do not align cleanly.
  • Test evidence exists but does not clearly support readiness decisions.
  • Critical knowledge is concentrated in one or two people.
  • Development process gates exist, but the resulting deliverables are inconsistent.

Method

How the service works

The service first establishes what your organisation expects to exist. Then reconstructs what currently exists and highlights where reality drifts.

1) NPI Reality Check

We review your stated NPI / V-model (or equivalent) to extract expected stages, deliverables, and governance assumptions. The output is a simple reference map used to align the as-built view.

2) System Archaeology

We reconstruct the system as it currently exists using the materials and deliverables already produced across the project (software, control logic, CAN, wiring/I/O, documentation, test notes). We identify hidden assumptions, inconsistencies across deliverables, and areas where evidence is missing.

3) Engineering judgement

Automated summarisation can help accelerate review. It cannot replace judgement. Findings are prioritised using practical systems engineering experience and a focus on delivery risk.

Inputs

What we work with

The service works with whatever information and materials are currently available. The aim is to build clarity from the current state and provide outputs you can act on.

Process / governance (NPI Reality Check)

  • NPI / V-model documents (or internal lifecycle description)
  • Gate definitions and expected deliverables
  • Templates: requirements, design, test, reviews (if used)
  • Change control / release approach (even if informal)
  • Definition of “done” / readiness criteria (if defined)

Technical materials (System Archaeology)

  • Control software / logic exports (as applicable)
  • CAN materials (DBC, spreadsheets, signal lists)
  • Wiring tables, I/O lists, pin maps
  • Architecture diagrams (even if out of date)
  • Test notes, commissioning notes, known issues list
  • Release notes / version history (if available)
Outputs

What you receive

Outputs are designed to be decision-grade: clear enough to brief leadership, onboard an engineer, and prioritise actions.

1) As-built System Reality Map

  • Functional behaviour and operating modes
  • Interfaces and dependencies (CAN, I/O, external subsystems)
  • Key assumptions annotated with evidence sources

2) Alignment Summary (process vs reality)

  • Expected deliverables vs available deliverables
  • Key inconsistencies and drift by lifecycle stage
  • Where governance assumptions are not met in practice

3) Ranked Risk Register

  • Technical, integration, and process risks
  • Severity / likelihood with confidence notes
  • Suggested mitigations and practical next actions

4) Recommendations & next steps

  • Immediate actions vs medium-term improvements
  • 30/60/90 day prioritised plan
  • Optional implementation support (engineering delivery)
